# v1.0 Migration Policy

This document describes how Trailmind handles schema and CLI changes between versions,
and what users can expect when upgrading within the 1.x line.

## Versioning

Trailmind follows [Semantic Versioning](https://semver.org/):

- **MAJOR** (1.x → 2.x): Breaking changes to core schema or CLI behavior
- **MINOR** (1.1 → 1.2): New features, backward-compatible
- **PATCH** (1.1.0 → 1.1.1): Bug fixes, no behavior changes

## Core Schema Stability (1.x)

Within the 1.x line, the following are considered **stable** and will not change
without a deprecation cycle:

### Frontmatter Fields

| Entity | Stable Fields |
|--------|---------------|
| Project | `slug`, `title`, `goal`, `state`, `owners`, `tags`, `created` |
| Epic | `slug`, `title`, `project`, `goal`, `state`, `start`, `target`, `roster`, `repos`, `created` |
| Task | `id`, `title`, `filer`, `owner`, `status`, `priority`, `created`, `start`, `due`, `code_paths`, `design_doc`, `depends_on`, `soft_depends_on`, `known_issues`, `deliverables`, `completed_deliverables` |
| Issue | `id`, `title`, `filer`, `owner`, `status`, `severity`, `created`, `linked_tasks`, `carried_into` |
| Milestone | `id`, `title`, `date`, `created` |
| Inbox | `item_id`, `title`, `author`, `status`, `created`, `resolved`, `resolver`, `note` |
| Spec | `title`, `status`, `created`, `scope`, `project`, `epic`, `linked_plans` |
| Plan | `title`, `status`, `created`, `scope`, `project`, `epic`, `linked_spec`, `generated_tasks` |

### Status Values

| Entity | Stable Values |
|--------|---------------|
| Task | `created`, `ready`, `in_progress`, `blocked`, `done`, `wontfix` |
| Issue | `open`, `done`, `wontfix` |
| Inbox | `open`, `resolved` |
| Project/Epic state | `planning`, `active`, `paused`, `completed`, `archived`, `cancelled` |
| Spec | `draft-for-review`, `approved-for-spec`, `superseded` |
| Plan | `draft`, `approved`, `in-progress`, `completed`, `superseded` |

### Priority Values

`low`, `medium`, `high`, `critical`

### Severity Values

`low`, `medium`, `high`, `critical`

## CLI Stability (1.x)

The following command patterns are stable:

- `trailmind status [--overview|--project|--epic]`
- `trailmind sweep [--project|--epic] [--stale-days] [--json]`
- `trailmind project init/list/set-status`
- `trailmind epic init/list/set-status`
- `trailmind task add/list/show/start/done/close/set-status/set-priority/due/assign`
- `trailmind issue add/list/show/close/carry/link/assign/set-severity`
- `trailmind milestone add/list/show`
- `trailmind inbox add/list/resolve/show`
- `trailmind plan spec init/list/show/set-status`
- `trailmind plan init/list/show/set-status/link-spec`
- `trailmind plan breakdown`
- `trailmind plan status`
- `trailmind task/issue pickup`
- `trailmind log`
- `trailmind stats [--json]`
- `trailmind doctor [--json]`
- `trailmind export/import`
- `trailmind init`
- `trailmind scan`
- `trailmind serve`

## Deprecation Policy

- Deprecated features are maintained for at least **one minor version** before removal.
- Deprecation warnings appear in CLI output and in `trailmind doctor`.
- The `CHANGELOG.md` documents all deprecations and removals.

## Migration Path

When upgrading:

1. Read `CHANGELOG.md` for the target version.
2. Run `trailmind doctor` to check for compatibility issues.
3. Run `trailmind task normalize-statuses --write` to fix legacy status values.
4. If a schema migration is needed, it will be documented in the migration guide.

## What Is NOT Stable (Yet)

The following may change within 1.x:

- HTML dashboard output format
- JSON output field additions (existing fields are stable)
- `trailmind serve` HTML templates
- Internal Python API (only CLI is user-facing)

## 2.0 Expectations

A 2.0 release would only be needed for:

- Changing the `projects/` directory structure
- Removing stable frontmatter fields
- Changing task status lifecycle
- Breaking CLI command names or argument patterns

The goal is to never need a 2.0 unless the core model fundamentally changes.
