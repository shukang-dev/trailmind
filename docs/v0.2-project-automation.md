# Trailmind v0.2 Project Automation

Trailmind v0.2 adds project automation helpers for task state, dependency
gates, deliverable tracking, inbox triage, and read-only sweep reports. The
workflow stays Markdown-backed: commands update repository files that can be
reviewed, diffed, and committed like any other project artifact.

All examples use public-safe demo names and `example.com` identities.

## Task Statuses

Tasks use these normalized statuses:

| Status | Meaning |
| --- | --- |
| `created` | The task exists but is not ready to start. |
| `ready` | The task is ready for work. |
| `in_progress` | Work is actively underway. |
| `blocked` | Work is paused on a blocker. |
| `done` | Work is complete. |
| `wontfix` | Work is intentionally closed without completion. |

`done` and `wontfix` are terminal statuses. Once a task reaches either status,
it cannot transition to another status.

Allowed transitions:

| From | To |
| --- | --- |
| `created` | `ready`, `in_progress`, `blocked`, `wontfix` |
| `ready` | `in_progress`, `blocked`, `done`, `wontfix` |
| `in_progress` | `ready`, `blocked`, `done`, `wontfix` |
| `blocked` | `ready`, `in_progress`, `wontfix` |
| `done` | none |
| `wontfix` | none |

## Status Normalization

Legacy task statuses can be audited and normalized:

```sh
trailmind task normalize-statuses
trailmind task normalize-statuses --write
```

The dry run reports legacy values without changing files. `--write` rewrites
recognized legacy statuses after validating task status data:

| Legacy status | Normalized status |
| --- | --- |
| `planned` | `created` |
| `integration` | `in_progress` |

## Status Changes

Use `task set-status` when a status change should include the actor and an
optional note in the task Activity Log:

These examples show alternative target statuses. Choose a target that is valid
for the task's current status.

```sh
trailmind task set-status T-123456-001 ready --actor alice --note "Ready to start."
trailmind task set-status T-123456-001 in_progress --actor alice --note "Started implementation."
trailmind task set-status T-123456-001 blocked --actor alice --note "Waiting for a dependency."
trailmind task set-status T-123456-001 done --actor alice --note "Verified acceptance criteria."
trailmind task set-status T-123456-001 wontfix --actor alice --note "Superseded by another task."
```

The command enforces the transition table above. Moving a task to `ready`,
`in_progress`, or `done` also checks hard dependencies. Moving a task to `done`
checks deliverables.

`task close` remains available for the common "finish this task" path:

```sh
trailmind task close T-123456-001 --closer alice --note "All deliverables are complete."
```

Closing a task uses the same transition, dependency, and deliverable gates as
other moves to `done`.

## Dependencies

Dependencies are declared when a task is created:

```sh
trailmind task add \
  --epic projects/demo_app/mvp \
  --filer alice@example.com \
  --owner alice@example.com \
  --title "Build Sign-In Form" \
  --code-paths "src/app.py,tests/test_app.py" \
  --depends-on T-123456-001 \
  --soft-depends-on T-123456-002 \
  --known-issues I-123456-001 \
  --deliverables "tests pass,docs updated"
```

Hard dependencies in `depends_on` must resolve to terminal tasks (`done` or
`wontfix`) before the dependent task can move to `ready`, `in_progress`, or
`done`.

Soft dependencies in `soft_depends_on` do not block the transition. When they
are not terminal, Trailmind prints a warning after the status change so the
caller can decide whether follow-up is needed.

Known issues in `known_issues` are task context. They do not block closure, but
`task close` reports linked open issues after the close command updates the
task.

## Deliverables

Deliverables are the task's explicit completion checklist. They can be set when
creating a task or added later:

```sh
trailmind task deliverable add T-123456-001 --item "docs updated" --actor alice
trailmind task deliverable complete T-123456-001 --item "docs updated" --actor alice
```

Trailmind normalizes whitespace in deliverable text. Completing a deliverable
requires the item to already be present in the task's `deliverables` list.
Adding deliverables to terminal tasks is rejected.

A task cannot move to `done` until every item in `deliverables` also appears in
`completed_deliverables`.

## Inbox

Project and epic inboxes capture notes that should not become tasks or issues
yet.

```sh
trailmind inbox add \
  --epic projects/demo_app/mvp \
  --author alice \
  --title "Review release checklist" \
  --note "Confirm the checklist before release."

trailmind inbox add \
  --project demo_app \
  --author alice \
  --title "Consider documentation cleanup" \
  --note "Review after the current milestone."
```

List open and resolved inbox items for a scope:

```sh
trailmind inbox list --epic projects/demo_app/mvp
trailmind inbox list --project demo_app
```

Resolve an inbox item after it has been handled:

```sh
trailmind inbox resolve IN-20260630-001 --resolver alice --note "Filed a follow-up task."
```

Inbox files live under official project or epic `inbox/` directories. Direct
path resolution is intentionally restricted to those locations.

## Sweep

`trailmind sweep` is a read-only report for project automation triage:

```sh
trailmind sweep
trailmind sweep --project demo_app
trailmind sweep --epic projects/demo_app/mvp
trailmind sweep --epic projects/demo_app/mvp --stale-days 14
```

The report includes:

- Ready tasks: non-terminal `created` and `ready` tasks without hard blockers.
- Blocked tasks: tasks with status `blocked` or unsatisfied hard dependencies.
- Stale tasks: non-terminal tasks whose latest created or Activity Log date is
  at least `--stale-days` old. The default is 7 days.
- Missing deliverables: non-terminal tasks that still have incomplete
  deliverables.
- Open inbox: unresolved inbox items under the selected scope.

Terminal tasks (`done` and `wontfix`) are omitted from sweep task sections.
